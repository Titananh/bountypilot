# P0.2 Packet 2 — Pinned approval and action-lifecycle contract

Status: approved design contract for tests and implementation. This file narrows Packet 2 of `p0.2-action-approval-execution-packets.md`; it does not widen the production allowlist.

Slice C implementation ambiguities are narrowed by
`p0.2-slice-c-lifecycle-contract.md`. That addendum may narrow this contract but
may not widen the production allowlist or effect surface.

## 1. Outcome and limits

Packet 2 builds a local, synchronous persistence boundary for action approval, claim, dispatch marking, finalization, lease recovery, human reconciliation, workflow events, and derived job state.

Packet 2 does not execute HTTP, browser, MCP, desktop, child-process, scan, exploit, report-submission, or other target effects. It does not claim exactly-once external delivery. It does not guarantee a finding is novel, accepted, or rewarded.

Production allowlist:

- `src/core/actions/action-queue.ts`
- `src/core/actions/action-review-store.ts`
- new `src/core/actions/action-approval-service.ts`
- new `src/core/actions/action-lifecycle.ts`
- `src/core/jobs/job-manager.ts`
- `src/core/jobs/workflow-event-store.ts`
- `src/cli/runtime.ts`, wiring only
- `src/stores/db/migrations.ts` and `src/stores/db/database.ts`, only to expose the
  existing `withImmediateTransaction` guard state through the read-only
  `hasImmediateTransaction()` helper described in section 5
- focused Packet 2 tests

Forbidden in Packet 2:

- `src/cli/index.ts`
- schema, migration SQL, migration ordering, migration normalization, or schema-version changes
- `src/core/actions/action-approval-context.ts`
- target-effect implementations
- test, lab, live-mode, caller-stack, or legacy approval bypasses
- autonomous retry, lease stealing, timers, scanning, reconciliation, or submission

## 2. Locked invariants

1. `ActionQueue.enqueue()` never defaults to `approved`.
2. Caller-supplied `status: "approved"` is rejected at runtime with
   `ACTION_APPROVAL_SERVICE_REQUIRED` before ID/time generation or any database write.
3. `ActionQueue.approve()` never performs `pending -> approved`; it fails with `ACTION_APPROVAL_SERVICE_REQUIRED` after preserving `ACTION_NOT_FOUND` for a missing ID.
4. Only `ActionApprovalService` may produce `approved`.
5. Every approved row has one structurally complete active review matching the action, job, reviewer, source, expiry, and all four hashes.
6. Both human and policy approvals have a finite expiry. Source values are exactly `human | policy`; the policy reviewer is exactly `system:policy-gate`.
7. Approval, claim, dispatch mark, finalize, expired-lease recovery, and human reconciliation each use a short synchronous transaction. No transaction crosses an `await`.
8. An execution token is a 32-byte CSPRNG bearer capability encoded as 64 lowercase hex. It is never returned by generic action reads or written to events, errors, reports, or logs.
9. Claim does not reclaim active or expired `running` rows.
10. A dispatch marker is written exactly once before an effect attempt.
11. Possible post-dispatch ambiguity becomes `outcome_unknown`; it is never auto-retried.
12. `outcome_unknown` requires a human attestation to resolve and reconciliation itself produces zero effects.
13. A job cannot become `completed` while any required action is pending, approved, running, blocked, failed, or outcome-unknown.
14. Non-required actions never block job completion.
15. Every composed write uses in-transaction store primitives; transaction-owning wrappers are never nested.
16. Transaction membership is determined only by the module-private guard owned by
    `withImmediateTransaction`; Packet 2 does not use `DatabaseSync.isTransaction`
    and therefore preserves the declared Node.js `>=22.13.0` runtime contract.

## 3. Public state and records

```ts
export type ActionStatus =
  | "pending"
  | "approved"
  | "running"
  | "executed"
  | "blocked"
  | "failed"
  | "outcome_unknown";

export type OutcomeCertainty =
  | "success"
  | "not_dispatched"
  | "possibly_dispatched";

export type EffectOutcome =
  | { kind: "success" }
  | { kind: "not_dispatched"; errorCode: string; errorMessage: string }
  | { kind: "possibly_dispatched"; errorCode: string; errorMessage: string };

export type ActionReviewDecision =
  | "approved"
  | "blocked"
  | "executed"
  | "failed";

export type ActionReviewSource = "human" | "policy";
```

`ActionRecord` maps every v2 lifecycle column except `execution_token`. It includes
`requiredForCompletion: boolean`, `activeReviewId`, all four planned hashes,
`executionOwner`, lease/start/dispatch/finish timestamps, outcome certainty,
redacted last-error fields, `updatedAt`, and `supersedesActionId`.

The SQLite `required_for_completion` and `requires_approval` values are each
decoded strictly: `0 -> false` and `1 -> true`. Any other stored value fails
closed with `ACTION_RECORD_INVALID`; neither is coerced with JavaScript
truthiness. This strictness is required before policy approval so a malformed
`requires_approval` value can never be interpreted as `false`.

`metadata_json` maps SQL `NULL` to absent metadata. Every non-null value must be
a nonempty valid JSON encoding of a non-null plain object; invalid JSON,
primitives, and arrays fail with fixed cause-free `ACTION_RECORD_INVALID` and
are never silently converted to missing metadata. Approval therefore cannot
authorize a weakened binding after stored metadata corruption.

For compatibility with pre-v2 terminal rows, `updatedAt` maps
`updated_at ?? executed_at ?? created_at`. No other lifecycle field is inferred:
null owner/lease/dispatch/finish/outcome/error columns remain absent on the
public record.

`executionToken` is available only on `ClaimedActionContext` returned to the executor owner.

All stored timestamps are canonical UTC ISO strings. All hash fields are 64-character lowercase hex.

## 4. Trusted dependencies and authority material

Program loading, binding-material resolution, and the approval clock are trusted
`ActionApprovalService` constructor dependencies. Review and workflow-event IDs
continue to come from the stores' trusted project `createId` generator; they are
not service dependencies and callers cannot supply them. Token generation is a
Slice C lifecycle dependency. Approval and lifecycle method callers cannot
supply clocks, IDs, hashes, job rows, policy decisions, capabilities, or
integration execution policy.

```ts
export interface ActionMaterialSource {
  action: {
    id: string;
    jobId: string;
    adapter: string;
    actionType: string;
    target: string | null;
    riskLevel: RiskLevel;
    requiresApproval: boolean;
    requiredForCompletion: boolean;
    metadata: Record<string, unknown>;
  };
  job: {
    id: string;
    type: string;
    target: string | null;
    mode: ExecutionMode;
  };
}

export type IntegrationBindingMaterial =
  | { kind: "builtin" }
  | {
      kind: "integration";
      policy: SafeIntegrationExecutionPolicy;
    };

export interface ResolvedBindingMaterial {
  normalizedTarget: string | null;
  capabilityEnforcement: CapabilityEnforcementInput;
  integration: IntegrationBindingMaterial;
}

export interface CurrentActionMaterial {
  program: LoadedProgram;
  scopeHash: string;
  policyHash: string;
  normalizedTarget: string | null;
  capabilityEnforcement: CapabilityEnforcementInput;
  integration: IntegrationBindingMaterial;
}

export interface ActionAuthorityDependencies {
  loadCurrentProgram(): LoadedProgram;
  resolveBindingMaterial(input: {
    source: ActionMaterialSource;
    program: LoadedProgram;
  }): ResolvedBindingMaterial;
}

export interface ActionApprovalServiceDependencies
  extends ActionAuthorityDependencies {
  now(): Date;
}
```

`ActionApprovalServiceDependencies.now()` is the only clock for the approval
review and action CAS. It is never called by `preview()`. On an approval path
that passes caller validation, read-only preflight, authority materialization,
the method-specific policy gate, and (for human approval) the context-token
comparison, it is called exactly once immediately before `BEGIN IMMEDIATE`.
The service copies `getTime()` once, requires a finite epoch, checked-adds
`ttlMs`, requires the result to be finite and within the JavaScript `Date`
range, and round-trips both instants through `toISOString()`. The resulting
`reviewedAt` is also the CAS `updatedAt`; `expiresAt` is strictly later.
Clock throw, invalid `Date`, overflow, or failed ISO round-trip maps to the fixed
cause-free `ACTION_APPROVAL_CONTEXT_UNAVAILABLE`. The event store and job
manager retain their own trusted internal timestamps. Approval does not accept
or generate an execution token; token generation is introduced in Slice C.

Malformed dependencies and any clock, program-loader, binding-resolver, or
authority-materialization exception are mapped to a fixed, cause-free
`ACTION_APPROVAL_CONTEXT_UNAVAILABLE` error. The original exception text and
authority material are never reflected to callers or persisted.

The service builds `ActionMaterialSource` itself from validated database rows;
missing targets become `null` and missing metadata becomes a fresh empty plain
object. That narrow projection is the resolver's entire row authority input.
The service canonicalizes it using schema
`approval-material-source/v1`, keeps the canonical string only in memory, and
requires exact canonical equality after the in-transaction reload before any
review insert. A mismatch is `ACTION_APPROVAL_RACE_LOST` for either approval
method. Status and lifecycle fields are excluded from the source projection
because they have separate exact preconditions before both comparisons.

Resolver output is snapshotted into deeply plain immutable data before the
transaction; no proxy, accessor, function, or mutable callback crosses the
boundary. `integration.kind = integration` requires a complete safe policy and
can never fall back to `builtin`; an unknown tag or malformed projection is
`ACTION_APPROVAL_CONTEXT_UNAVAILABLE`. The trusted resolver must derive the tag
from the current integration registry. This tagged union removes the ambiguous
meaning of a nullable execution policy.

The service itself must create a fresh `ScopeGuard` and `PolicyGate`, construct
the current `PolicyActionInput`, evaluate the current decision, build the
authority snapshot, validate capability/action/risk/mode/target consistency,
and compute `actionHash` and `contextHash`.

The service constructs `PolicyActionInput` exactly from current rows/material:
job mode, action type, normalized target when present, action risk,
`capabilityEnforcement.stateChanging`, `.destructive`, `.id` as capability,
`.requiresApprovalByDefault`, and current program `rules.lab_mode`. Capability
ID and action type must be canonical nonempty strings; capability action type
and risk must exactly equal the action; and `allowedModes` must be a valid set
of supported modes. When `requiresTarget` is true, both raw and normalized targets are
required. A present raw target is normalized by the fresh `ScopeGuard` and must
exactly equal resolver `normalizedTarget`; an absent target requires
`normalizedTarget === null`. Malformed or inconsistent material is context
unavailable.

When `requiresScope` is true, a valid normalized target denied by the fresh
scope guard produces an effective `block`; it is not materialization failure.
For `integration` material, policy name must equal action adapter and timeout
must be a safe integer in `1..120_000`. A structurally valid projection with
`enabled` or `allowExecute` false, action type absent from `capabilities`, action
type or capability ID present in `blockedCapabilities`,
`capabilityEnforcement.blockedByDefault === true`, current job mode absent from
`allowedModes`, or denied required scope produces effective decision `block`.
Invalid
types, hashes, entrypoints, arrays, timeout, linkage, or an unknown integration
is context unavailable. Enforcement blocks take precedence over the fresh
`PolicyGate` result and use fixed public reason
`Blocked by current authority enforcement`; otherwise the `PolicyGate` decision
and reason are used unchanged.

An unknown capability, unknown integration tag, or `integration` material
without a complete current safe execution-policy projection fails closed.
Reclassifying it as `builtin` is forbidden.

Every approval uses this exact two-load sequence:

1. Validate and snapshot caller input without invoking accessors.
2. Read the action and linked job outside a transaction and reject missing,
   terminal, non-`pending`, or dirty state.
3. Build and canonicalize `ActionMaterialSource`, call
   `loadCurrentProgram()` once, then
   `resolveBindingMaterial({ source, program })` once. Build a fresh
   authority snapshot, `ScopeGuard`, `PolicyGate`, policy action/result, and
   immutable binding-material projection. Nothing is reused from
   `runtime.config` or a previous preview/approval.
4. Apply the method-specific decision gate and human token comparison.
5. Call and validate the approval clock as described above.
6. Enter `withImmediateTransaction`, reload action and job, and recheck every
   precondition. Rebuild and canonicalize `ActionMaterialSource`; exact
   inequality with step 3 is `ACTION_APPROVAL_RACE_LOST`. Then rebuild the
   policy action, action binding, `actionHash`, and `contextHash` from the
   reloaded rows plus the immutable program/authority and binding projection
   from step 3. The loader, resolver, and clock are never called in the
   transaction.

`preview()` performs steps 1--3 and the pure hash computation only. It neither
calls the clock nor begins a transaction.

Residual race: program authority can change after materialization and before the eventual request. Packet 2 pins the materialized hashes to the action, review, claim context, and structured event metadata. P0.3 must revalidate current authority at the request boundary before any target traffic.

No filesystem, network, browser, MCP, or process operation occurs inside the transaction.

## 5. Workflow event store

```ts
class WorkflowEventStore {
  record(input: WorkflowEventInput): WorkflowEventRecord;
  recordInTransaction(input: WorkflowEventInput): WorkflowEventRecord;
}

// Internal transaction-state API exported by the database module. It reads the
// same module-private WeakSet used by withImmediateTransaction.
function hasImmediateTransaction(db: BountyDatabase): boolean;
```

- `record()` owns `withImmediateTransaction`.
- `recordInTransaction()` requires `hasImmediateTransaction(db) === true` and never begins or commits.
- `hasImmediateTransaction()` is a pure guard-state read. It performs no SQL,
  never begins/commits/rolls back, and is true only while that exact handle is
  executing the synchronous `withImmediateTransaction` callback. Guard cleanup
  remains mandatory on every commit, rollback, begin failure, callback failure,
  thenable rejection, and commit failure path.
- `DatabaseSync.isTransaction` is forbidden here: it was introduced after the
  project's minimum Node.js version and would silently make the published
  `>=22.13.0` engine declaration false.
- `MAX(sequence) + 1` and the insert occur under the same `BEGIN IMMEDIATE` lock.
- Message and metadata are masked before persistence and again on read.
- A unique-sequence collision is an invariant failure and rolls back. It is not retried and is never treated as an idempotent match to a different event.
- Event status remains the existing compatible union. Reconciliation uses status `paused` with `reasonCode: "reconciliation_required"`; it does not invent an `outcome_unknown` event status.

## 6. Review store

The read model exposes every v2 review column as optional because legacy non-active rows may contain nulls.

```ts
export interface ActionReviewRecord {
  id: string;
  actionId: string;
  jobId?: string;
  decision: ActionReviewDecision;
  note?: string;
  createdAt: string;
  reviewerId?: string;
  source?: ActionReviewSource;
  reviewedAt?: string;
  expiresAt?: string;
  scopeHash?: string;
  policyHash?: string;
  actionHash?: string;
  contextHash?: string;
  invalidatedAt?: string;
  invalidationReason?: string;
}
```

For an active approval, source and reviewer are structurally linked:

- `source = policy` requires `reviewer_id = system:policy-gate`;
- `source = human` requires a validated nonempty reviewer other than
  `system:policy-gate`.

The full active-approval read predicate also requires a concrete nonempty action
job ID, exact review action/job linkage, canonical `reviewedAt` and `expiresAt`
with `expiresAt > reviewedAt`, and four matching hashes on the action and review.
Every hash must be 64 lowercase hexadecimal characters and must not be the
all-zero sentinel. A matching-but-malformed value is invalid. Reviewer IDs are
trimmed and persisted in canonical form, contain 1..256 Unicode code points,
and reject U+0000..U+001F plus U+007F. A canonical timestamp is exactly the
value produced by `Date#toISOString()` (`YYYY-MM-DDTHH:mm:ss.sssZ`); parsing
must succeed and round-trip to the identical string.
Any approved row that violates this predicate makes generic `get()` and `list()`
fail closed with `ACTION_APPROVAL_INVALID`.

`CompleteApprovalReviewInput` has no caller-supplied review ID. The store
generates the trusted ID with the project ID generator. An own `id` property
supplied through a runtime cast is invalid (`ACTION_APPROVAL_INVALID`) and
writes no row. The input requires a
nonempty `actionId`, concrete nonempty `jobId`, `decision: "approved"`, the linked reviewer/source pair, canonical
`reviewedAt`, an `expiresAt` strictly later than `reviewedAt`, and all four
nonzero 64-character lowercase-hex hashes. The store sets
`created_at === reviewed_at`, masks `note` before persistence and again on read,
and preserves non-secret text. Comparison to the injected current clock remains
the approval service's responsibility. Invalid input fails with
`ACTION_APPROVAL_INVALID` and writes no row.

Transaction-bound primitives:

```ts
interface InvalidateApprovalResult {
  readonly changes: 0 | 1;
}

insertApprovalInTransaction(input: CompleteApprovalReviewInput): ActionReviewRecord;
invalidateApprovalInTransaction(input: InvalidateApprovalInput): InvalidateApprovalResult;
insertReconciliationInTransaction(input: ReconciliationReviewInput): ActionReviewRecord;
```

An approval insert requires all structural fields, including non-null expiry and all four hashes. An invalidation is a one-time CAS on both `invalidated_at IS NULL` and `invalidation_reason IS NULL`. It returns only the frozen normalized row-change count; it does not select or return the review row. Expiry and hash-drift callers require `changes = 1`; current-authority block repair accepts `0 | 1` because a missing, mismatched, or already-invalidated review must not prevent the action from being blocked.

Reconciliation safely reuses the current table because the schema has no decision CHECK and the active-review predicate only accepts `decision = 'approved'`. A reconciliation row must:

- have `decision = executed | failed`;
- have `source = human`;
- have a nonempty validated reviewer ID and nonempty redacted attestation in `note`;
- match action and job;
- copy the four planned hashes from the original approval;
- set `created_at === reviewed_at`;
- remain non-active: `actions.active_review_id` continues to point to the original approval review;
- never authorize claim or effect;
- never use `system:policy-gate` as reviewer.

`expires_at` is not used to authorize a reconciliation record. It may be null because the record is non-active and the structural active-approval predicate is not applied to it.

## 7. Approval service

```ts
export interface ActionApprovalChallenge {
  actionId: string;
  jobId: string;
  policyDecision: PolicyDecision;
  policyReason: string;
  scopeHash: string;
  policyHash: string;
  actionHash: string;
  contextHash: string;
}

export interface HumanApprovalInput {
  actionId: string;
  reviewerId: string;
  expectedContextHash: string;
  ttlMs: number;
  note?: string;
}

export interface PolicyApprovalInput {
  actionId: string;
  ttlMs: number;
  note?: string;
}

export interface ActionApprovalResult {
  action: ActionRecord;
  review: ActionReviewRecord;
  event: WorkflowEventRecord;
  job: JobRecord;
}

export class ActionApprovalService {
  constructor(
    db: BountyDatabase,
    dependencies: ActionApprovalServiceDependencies,
  );

  preview(actionId: string): ActionApprovalChallenge;
  approveHuman(input: HumanApprovalInput): ActionApprovalResult;
  approvePolicy(input: PolicyApprovalInput): ActionApprovalResult;
}
```

The service constructs its queue, review, event, and job stores on the exact
constructor database handle. Callers cannot inject a store backed by a different
handle. Every returned `ActionApprovalResult` is composed from the records
produced inside the transaction that successfully commits; it contains no
execution token or raw authority material.

`preview()` is strictly read-only. It validates the action identifier, reloads
the action and its concrete nonterminal job, requires the same `pending` and
clean execution/review state used by approval step 2, and freshly materializes
the current program and authority context on every call. It returns the current
policy decision even when that decision is `block` or `require_approval`; this
lets a caller inspect the challenge but grants no authority. Each approval
method independently reloads and rematerializes everything again and treats the
preview result only as an untrusted context token supplied through
`expectedContextHash`.

Preflight error order is exact for preview and both approval methods: a missing
action uses existing `ACTION_NOT_FOUND`; a non-`pending` action uses
`ACTION_APPROVAL_NOT_PENDING`; a null/trim-empty action `jobId` uses
`ACTION_APPROVAL_JOB_REQUIRED`; a referenced missing job uses existing
`JOB_NOT_FOUND`; job status `failed | completed` uses
`ACTION_APPROVAL_JOB_TERMINAL`; any dirty review/lifecycle/execution/error field
uses `ACTION_APPROVAL_STATE_INVALID`.
Clean state means `executedAt`, `activeReviewId`, all four planned hashes, the
hidden execution token, `executionOwner`, `leaseExpiresAt`, `startedAt`,
`dispatchStartedAt`, `finishedAt`, `outcomeCertainty`, `lastErrorCode`, and
`lastErrorMessage` are all absent/SQL `NULL`. Existing malformed-row errors such
as `ACTION_RECORD_INVALID` propagate unchanged.

All public caller-input validation is a pure snapshot before any database read,
program loading, resolver or clock call, ID generation, or write. Human and
policy DTOs must be non-null plain/null-prototype records whose required fields
are own data properties; accessors, hostile reflection, symbol fields, and
unknown own fields fail with fixed cause-free `ACTION_APPROVAL_INVALID` without
invoking caller code. `preview` and DTO `actionId` values must be strings equal
to their trim, contain 1..256 Unicode code points, and contain no C0/DEL control
characters; invalid values use `ACTION_APPROVAL_INVALID`.

When multiple caller fields are invalid, validation precedence is: envelope,
unknown/accessor fields, `actionId`, and `note` -> `ACTION_APPROVAL_INVALID`;
then human reviewer -> `ACTION_APPROVAL_REVIEWER_INVALID`; then TTL ->
`ACTION_APPROVAL_TTL_INVALID`; then human context shape ->
`ACTION_APPROVAL_CONTEXT_INVALID`. Policy input omits the reviewer/context
steps. All caller validation completes before the first database read.

`ttlMs` is a required safe integer in `1..86_400_000`; otherwise
`ACTION_APPROVAL_TTL_INVALID`. Human reviewer IDs use the review store's exact
rule: the raw value contains no C0/DEL, its trimmed value has 1..256 Unicode
code points, and the canonical result is not `system:policy-gate`; otherwise
`ACTION_APPROVAL_REVIEWER_INVALID`. Optional `note` is absent, `undefined`, or a
string; any other shape is `ACTION_APPROVAL_INVALID`. The review store remains
responsible for trimming, empty-to-null conversion, and write-time secret
masking; neither service errors nor events include the note.

The human context token is exactly 64 lowercase hex; wrong type/shape is
`ACTION_APPROVAL_CONTEXT_INVALID`. A syntactically valid token, including the
all-zero sentinel, that differs from the freshly recomputed nonzero context
hash is `ACTION_APPROVAL_CONTEXT_STALE`.

The decision matrix is exact and is re-applied inside the transaction after
hash recomputation:

- current decision `block` rejects both methods with
  `ACTION_APPROVAL_POLICY_BLOCKED`;
- `approveHuman` accepts current `allow` or `require_approval` after a matching
  context token, persists source `human`, and uses the validated human reviewer;
- `approvePolicy` accepts only current `allow` with
  `action.requiresApproval === false`, persists source `policy` with reviewer
  exactly `system:policy-gate`, and otherwise rejects with
  `ACTION_APPROVAL_HUMAN_REQUIRED`.

Malformed loader/resolver output, unknown capabilities, unsupported raw
mode/risk/decision values, action/capability/action-type/risk/mode/target
inconsistency, or an incomplete execution-capable integration projection maps
to fixed cause-free `ACTION_APPROVAL_CONTEXT_UNAVAILABLE`. A valid current
policy `block` is not a materialization fault and uses the decision-matrix error
above.

Approval transaction order:

1. Reload action and job.
2. Require `pending`, a concrete nonterminal job, null active review, and every
   clean lifecycle/error/execution field.
3. Rebuild `ActionMaterialSource`, require exact canonical equality with the
   preflight source or throw `ACTION_APPROVAL_RACE_LOST`, then rebuild and
   validate the current policy action, decision, action binding, and all four
   hashes from the reloaded rows and immutable material.
4. Re-apply the exact decision matrix; for human approval, compare the reviewer
   context token again.
5. Insert the complete immutable review with the single captured clock instant.
6. CAS `pending -> approved`, set active review and all four planned hashes, and
   include every clean-state predicate.
7. Append `action-review/completed` event with action ID, review ID, source, and
   pinned context hash only.
8. Recompute the job using `finalizeInTransaction`.
9. Return the four callback records from the transaction callback and commit.

Any failure rolls back review, action, event, and job together. The public
result is returned only after commit succeeds. `BEGIN`, commit, rollback,
database, and already-typed store/transaction errors propagate unchanged; they
are not remapped to `ACTION_APPROVAL_CONTEXT_UNAVAILABLE`, and no partial result
object is returned.

The event has phase `action-review`, status `completed`, and fixed message
`Action approval completed`. Its metadata object has exactly the keys
`actionId`, `reviewId`, `source`, and `contextHash`; note, target, action
metadata, program paths, and other authority material are forbidden.

The token-free approval-candidate reload is owned by `ActionQueue`:

```ts
export type PendingApprovalCandidate = ActionRecord & {
  jobId: string;
  status: "pending";
};

ActionQueue.requireCleanPendingForApproval(
  actionId: string,
): PendingApprovalCandidate;
```

This method is transaction-neutral, never begins or commits, reloads on every
call, and is usable both by `preview()` and inside the tracked approval
transaction. It first applies the same canonical action-ID rule as the service;
invalid input is fixed `ACTION_APPROVAL_INVALID` before SQL. It uses the public token-free action projection plus a computed
SQL boolean covering every clean field, including
`execution_token IS NULL`; the bearer value is never selected, materialized in
JavaScript, or returned. Approval requires raw stored status exactly `pending`;
the compatibility-only raw `planned` value is `ACTION_APPROVAL_NOT_PENDING`
even though generic reads normalize it. The method applies the preflight error
order above. The final
CAS still repeats every predicate, so this reload is not the concurrency
boundary.

The action CAS in step 6 is owned by an `ActionQueue` transaction primitive,
not by ad-hoc service SQL:

```ts
export interface ApproveActionWithReviewInput {
  actionId: string;
  jobId: string;
  reviewId: string;
  scopeHash: string;
  policyHash: string;
  actionHash: string;
  contextHash: string;
  updatedAt: string;
}

ActionQueue.approveWithReviewInTransaction(
  input: ApproveActionWithReviewInput,
): ActionRecord;
```

The primitive's first executable operation is the pure same-handle tracked
transaction guard. Outside that transaction it throws fixed cause-free
`ACTION_QUEUE_TRANSACTION_REQUIRED` before input reflection or SQL. Inside the
transaction it snapshots a non-null plain/null-prototype input with exactly the
pinned own data fields and rejects accessors, hostile reflection, symbol fields,
and unknown fields. Each of `actionId`, `jobId`, and `reviewId` must be a string
equal to its trim, contain 1..256 Unicode code points, and contain no C0/DEL
control characters. It also requires four nonzero lowercase hex-64 hashes and a
finite canonical UTC ISO `updatedAt`; any violation is fixed cause-free
`ACTION_APPROVAL_INVALID`.

It then explicitly selects the exact review columns needed for authorization,
never `note`, and validates the review visible in that transaction: exact
ID/action/job linkage; decision `approved`; both invalidation fields `NULL`;
source/reviewer exactly following the human/policy rule; canonical
`created_at === reviewed_at === input.updatedAt`; canonical
`expires_at > reviewed_at`; and the same four nonzero hashes as the input.
Missing or malformed linkage is `ACTION_APPROVAL_INVALID`. With the public
signature and frozen schema this proves that the exact complete review is
visible, not its historical transaction of origin; the service's insert-before-
CAS ordering and exclusive approval architecture are separate invariants.

The primitive performs exactly one `UPDATE actions`: it sets only status
`approved`, active review ID, the four planned hashes, and `updated_at`; its
predicate repeats exact action/job IDs, status `pending`, all 15 SQL-null clean
fields (`executed_at`, `active_review_id`, four planned hashes,
`execution_token`, `execution_owner`, `lease_expires_at`, `started_at`,
`dispatch_started_at`, `finished_at`, `outcome_certainty`,
`last_error_code`, `last_error_message`), plus an `EXISTS` check for the same
stable review linkage and hashes. `changes !== 1` is fixed
`ACTION_APPROVAL_RACE_LOST`; there is no retry or idempotent-success branch.
It reloads through the public token-free projection and returns that action.
An impossible missing post-update row is `ACTION_RECORD_INVALID`; structural
`ACTION_APPROVAL_INVALID` and other already-typed store/database/transaction
errors otherwise propagate.

## 8. Lifecycle API

```ts
export class ActionLifecycle {
  claim(input: {
    actionId: string;
    executionOwner: string;
    leaseMs?: number;
  }): ClaimedActionContext;

  markDispatch(input: {
    actionId: string;
    executionToken: string;
    executionOwner: string;
  }): { dispatchStartedAt: string };

  finalize(input: {
    actionId: string;
    executionToken: string;
    executionOwner: string;
    outcome: EffectOutcome;
  }): ActionRecord;

  recoverExpiredLease(actionId: string):
    | { kind: "active" | "not_running" }
    | { kind: "recovered"; status: "failed" | "outcome_unknown" };

  reconcileOutcome(input: {
    actionId: string;
    reviewerId: string;
    attestation: string;
    resolution: "effect_confirmed" | "no_successful_effect_confirmed";
  }): {
    action: ActionRecord;
    reconciliationReviewId: string;
    status: "executed" | "failed";
  };
}
```

Default lease is 60 seconds. Valid leases are positive safe integers and at most one hour. Lease expiry is `min(now + leaseMs, review.expiresAt)`. An already expired review is invalidated; it never produces a zero-length lease.

### 8.1 Claim

Claim validates, in order:

1. caller input;
2. action existence;
3. current action status;
4. concrete matching nonterminal job and clean lifecycle fields;
5. definitive current scope/policy block;
6. structurally complete active review;
7. review expiry;
8. scope, policy, action, then context hash drift;
9. CAS conflict.

The execution token is `randomBytes(32).toString("hex")`, or an injected equivalent in tests. A partial-unique-index collision is fatal `ACTION_TOKEN_COLLISION`; no retry occurs.

The claim CAS includes action ID, `status = approved`, active review ID, null token/owner/lease, and exact planned hashes. It writes `running`, token, owner, lease, `started_at`, clean dispatch/final fields, and `updated_at`. The running event and job derivation are in the same transaction.

For expiry or drift, the transaction returns a tagged rejection instead of throwing:

1. invalidate the review once;
2. CAS action back to `pending` and clear active review;
3. append the paused/drift event;
4. derive job state;
5. commit;
6. throw the domain `BountyPilotError` only after `withImmediateTransaction` returns successfully.

If commit fails, the commit error propagates and no domain rejection is thrown.

A definitive current scope or policy block invalidates the review and moves the action to `blocked` atomically instead of demoting it.

### 8.2 Dispatch mark

`markDispatch` requires a running row, exact token, exact owner, a well-formed active lease, the exact stored lease value, a null dispatch marker, and a still-valid active approval review. It validates expiry numerically before the CAS. A second call throws `ACTION_DISPATCH_ALREADY_MARKED`; it never returns a second permit.

### 8.3 Finalize

Mapping:

- `success` requires a dispatch marker and active lease -> `executed`, certainty `success`, set both `executed_at` and `finished_at`;
- `not_dispatched` with no marker -> `failed`, certainty `not_dispatched`;
- `not_dispatched` after a marker -> coerce to `outcome_unknown`, certainty `possibly_dispatched`;
- `possibly_dispatched` -> `outcome_unknown`, certainty `possibly_dispatched`.

Untyped Packet 3 exceptions must be mapped by the executor to `possibly_dispatched`.

Finalize CAS includes action ID, running status, exact token, exact owner, and exact stored lease. Wrong token/owner/status/lease makes no successful write. Error codes use a bounded safe format; error messages are masked and length-limited before persistence.

Terminalization clears token, owner, and lease but retains approval pointer and audit timestamps. Event and job derivation share the transaction.

### 8.4 Expired-lease recovery

Recovery is explicit and local; it is never a timer or retry.

- Non-running -> no-op.
- Active well-formed lease -> no-op.
- Expired, well-formed lifecycle with no dispatch marker -> `failed/not_dispatched`.
- Dispatch marker present, malformed/missing token/owner/lease, or inability to prove pre-dispatch -> `outcome_unknown/possibly_dispatched`.
- Repeated recovery -> no-op.
- Recovery never returns the action to pending/approved and never invokes an effect.

### 8.5 Human reconciliation

Only a validated human reviewer with a nonempty redacted attestation may reconcile `outcome_unknown`.

- `effect_confirmed` -> `executed/success`.
- `no_successful_effect_confirmed` -> `failed/possibly_dispatched`; the human resolves the action outcome but the original transport ambiguity remains preserved in `outcome_certainty`.

The transaction inserts the non-active reconciliation review, CASes `outcome_unknown` to the resolved status, appends an event, derives the job, and commits. Exactly one concurrent reconciliation wins. It performs zero effects.

## 9. Job derivation

`JobStatus` remains `queued | running | paused | failed | completed`.

```ts
export interface RequiredActionCounts {
  approved: number;
  blocked: number;
  executed: number;
  failed: number;
  outcome_unknown: number;
  pending: number;
  running: number;
}

export interface JobRecord {
  // existing fields omitted here
  pauseReason: PauseReason | null;
  statusDetail: string | null;
}

export class JobManager {
  finalize(jobId: string): JobRecord;
  finalizeInTransaction(jobId: string): JobRecord;
}
```

`finalize()` owns `withImmediateTransaction`. `finalizeInTransaction()` requires
`hasImmediateTransaction(db) === true`, never opens or closes a transaction,
reloads the job, derives from rows visible in that transaction, and updates the
job only when the derived tuple changed. Calling it outside the helper fails
with `JOB_TRANSACTION_REQUIRED`.

Pause reasons:

- `approval_required`
- `execution_ready`
- `policy_drift`
- `policy_blocked`
- `reconciliation_required`
- `budget_exhausted`
- `manual_review`

Mixed required-action precedence, first match wins:

1. any `outcome_unknown` -> `paused/reconciliation_required`;
2. else any `running` -> `running`;
3. else any `blocked | failed` -> `failed`;
4. else any `pending` -> `paused/approval_required`;
5. else any `approved` -> `paused/execution_ready`;
6. else all required executed, or zero required -> `completed`.

This order never hides ambiguous external effect state and never labels a job terminal while another required action is actively running.

The derived pause reason is `reconciliation_required` for outcome-unknown,
`approval_required` for pending, and `execution_ready` for approved. A failed
job uses `policy_blocked` when at least one required action is blocked and null
when the failing set contains only `failed`; running and completed jobs use
null.

After every successful finalization, `status_detail` is non-null canonical JSON
produced by `canonicalize(...).toString("utf8")` over exactly:

```json
{
  "schemaVersion": 1,
  "requiredActionCounts": {
    "approved": 0,
    "blocked": 0,
    "executed": 0,
    "failed": 0,
    "outcome_unknown": 0,
    "pending": 0,
    "running": 0
  }
}
```

Every value is the nonnegative integer count of actions with
`required_for_completion = 1` in that status. All seven keys are always
present; no other key is allowed. This makes the representation deterministic,
count-only, and incapable of containing an action ID, target, metadata,
reviewer note, token, or secret.

Non-required actions are excluded from both the decision and these counts.
Direct `updateStatus(id, "completed")` uses a transaction and the same derivation:
if any required action is not `executed`, it throws `JOB_COMPLETION_BLOCKED`
before writing the job; otherwise it delegates to `finalizeInTransaction()`.
Thus legacy jobs with zero required actions remain completable, while direct
completion can never bypass a required action.

## 11. RED-test slices and writer order

Use one writer at a time. `bp-test-writer` is the only role allowed to edit tests; `bp-implementer` never edits tests.

### Slice A — storage, fail-closed state, and transaction composition

Tests first, using only modules that already exist:

1. enqueue defaults to pending, including low-risk actions;
2. explicit approved enqueue and direct `approve()` from every existing action
   state fail closed, while a missing action remains `ACTION_NOT_FOUND`;
3. malformed approved rows table-test every active-review predicate component
   on both `get` and `list`, including the exact policy reviewer/source link;
4. action records map every v2 field, decode `requiredForCompletion` to a
   boolean, preserve legacy terminal timestamps, and never expose a token from
   either `get` or `list`;
5. complete approval review insert requires an existing transaction, rejects
   every invalid structural field with zero writes, persists every v2 field,
   and proves write/read redaction without dropping non-secret note text;
6. event `recordInTransaction` requires a transaction, allocates sequence under
   it, composes with caller rollback, and maps a forced unique collision to
   `WORKFLOW_EVENT_SEQUENCE_CONFLICT` without retry;
7. two truly concurrent public `record()` wrappers on separate handles produce
   unique monotonic sequences;
8. event write-time redaction remains effective through raw SQL and returned
   records contain `[REDACTED]` while preserving public message/metadata fields;
9. both job finalizer methods are tested: the public wrapper owns a transaction,
   the primitive rejects outside one and composes with rollback inside one;

## 12. Compatibility truth

The old suite and production call sites rely on implicit approval and direct lifecycle mutation. They cannot all retain their old live behavior while this invariant is enforced. Do not hide the conflict with a bypass or synthetic review.

Unmigrated live surfaces are not considered safe. They must either remain planning-only/fail-closed or be migrated in Packets 3–6. P0.2 is not complete until Packet 7 proves all target-effect call sites route through `ActionExecutor` and the full regression/safety/build/smoke gates pass.
