# P0.2 Packet 2 Slice C lifecycle contract

Status: implementation lock for the ambiguities left open by
`p0.2-packet-2-contract.md` sections 8-10. The parent contract remains
authoritative. This addendum may only narrow behavior; it may not add an effect,
network call, retry, lease steal, auto-reconciliation, or approval bypass.

## 1. Public API and trusted dependencies

```ts
export type EffectOutcome =
  | { kind: "success" }
  | { kind: "not_dispatched"; errorCode: string; errorMessage: string }
  | { kind: "possibly_dispatched"; errorCode: string; errorMessage: string };

export interface ClaimedActionContext {
  action: ActionRecord;
  executionToken: string;
}

export interface ActionLifecycleDependencies
  extends ActionAuthorityDependencies {
  now(): Date;
  generateExecutionToken(): string;
}

export class ActionLifecycle {
  constructor(
    db: BountyDatabase,
    dependencies: ActionLifecycleDependencies,
  );

  claim(input: {
    actionId: string;
    executionOwner: string;
    leaseMs?: number;
  }): ClaimedActionContext;

  markDispatch(input: {
    actionId: string;
    executionToken: string;
    executionOwner: string;
  }): { dispatchStartedAt: string };

  finalize(input: {
    actionId: string;
    executionToken: string;
    executionOwner: string;
    outcome: EffectOutcome;
  }): ActionRecord;

  recoverExpiredLease(actionId: string):
    | { kind: "active" | "not_running" }
    | { kind: "recovered"; status: "failed" | "outcome_unknown" };

  reconcileOutcome(input: {
    actionId: string;
    reviewerId: string;
    attestation: string;
    resolution: "effect_confirmed" | "no_successful_effect_confirmed";
  }): {
    action: ActionRecord;
    reconciliationReviewId: string;
    status: "executed" | "failed";
  };
}
```

The lifecycle constructs `ActionQueue`, `ActionReviewStore`,
`WorkflowEventStore`, and `JobManager` on the exact constructor database
handle. Callers cannot inject stores, rows, hashes, decisions, IDs, timestamps,
or jobs.

The runtime dependency uses `randomBytes(32).toString("hex")`. Tests may inject
an equivalent generator. A generated token must be a 64-character lowercase
hexadecimal string. All-zero is syntactically valid because token values, unlike
authority hashes, have no sentinel value. Generation throw or malformed output
becomes a fixed, cause-free `ACTION_TOKEN_INVALID` before `BEGIN IMMEDIATE`.

## 2. Caller boundary, clocks, and secret handling

Every method snapshots its public DTO before SQL, dependency calls, clock
calls, ID generation, or writes. Envelopes must be exact non-null plain or
null-prototype records with required own data fields. Reject Proxy/revoked
Proxy before reflection. Reject symbols, accessors, missing/unknown fields, and
custom prototypes without invoking caller code.

- `actionId`: primitive string equal to trim, 1..256 Unicode code points, no
  C0/DEL. Generic envelope/action-ID failure is `ACTION_LIFECYCLE_INVALID`.
- `executionOwner`: raw string contains no C0/DEL; trimmed value has 1..256
  Unicode code points. Failure is `ACTION_OWNER_INVALID`.
- `executionToken`: lowercase hex-64. Shape failure is
  `ACTION_TOKEN_INVALID`; a well-formed token unequal to storage is
  `ACTION_DISPATCH_TOKEN_MISMATCH`.
- `leaseMs`: absent means 60,000. Otherwise it is a safe integer in
  `1..3_600_000`. Failure is `ACTION_LEASE_INVALID`.
- `EffectOutcome`: exact tagged union. Error code is a primitive matching
  `^[A-Z][A-Z0-9_]{0,63}$`; error message is a primitive string of at most
  2,000 Unicode code points. Failure is `ACTION_OUTCOME_INVALID`.
- Reconciliation reviewer follows the human approval reviewer rule and cannot
  be `system:policy-gate`; failure is
  `ACTION_RECONCILIATION_REVIEWER_INVALID`.
- Reconciliation resolution must be one of the two exact literals; failure is
  `ACTION_RECONCILIATION_RESOLUTION_INVALID`.

Each state-mutating path copies `dependencies.now()` exactly once outside the
transaction and validates the `Date` slot and canonical UTC ISO round-trip.
Claim may perform its one token-generation call after this validation; it then
immediately enters `withImmediateTransaction`. Clock failure occurs before token
generation and is the fixed,
cause-free `ACTION_APPROVAL_CONTEXT_UNAVAILABLE`. Store, database, transaction,
BEGIN, COMMIT, and ROLLBACK errors are never remapped by the lifecycle.

Untrusted error messages and attestations are processed by a lifecycle-local
text sanitizer: `maskSecrets`, then replace every standalone lowercase hex-64
substring with `[REDACTED]`, then trim and bound the result. Do not change the
global secret masker because structured scope/policy/action/context hashes are
valid audit data. No event contains error text or attestation. No SQL SELECT
returns `execution_token` to JavaScript except no path at all: claim selects
only `execution_token IS NULL`; dispatch/finalize select only a bound-token
equality boolean; recovery selects only null/shape booleans.

## 3. Shared authority materialization

Approval and lifecycle must share one internal authority materializer and one
pure challenge recomputation function. Do not copy the approval hashing or
policy pipeline into a second implementation.

Outside the transaction, lifecycle claim:

1. loads a narrow token-free action/job source;
2. calls `loadCurrentProgram()` once;
3. calls `resolveBindingMaterial()` once;
4. snapshots deeply plain immutable program/resolver material;
5. derives current scope, policy, action, and context hashes.

Inside the transaction, it reloads the source and recomputes from the immutable
material without loader, resolver, clock, token generator, filesystem,
network, browser, MCP, process, or effect calls.

The shared materializer also returns an internal block class:

- a current `ScopeGuard` denial is `scope`;
- a `PolicyGate`, capability, or integration enforcement block is `policy`;
- otherwise it is `null`.

This value is never exposed in the public approval challenge.

## 4. Claim

Status precedence after valid caller input and action existence:

1. `pending` or compatibility `planned` -> `ACTION_NOT_APPROVED`;
2. `running` with active or expired lease -> `ACTION_LEASE_HELD`;
3. `outcome_unknown` -> `ACTION_RECONCILIATION_REQUIRED`;
4. `executed | blocked | failed` -> `ACTION_TERMINAL`;
5. unknown storage -> `ACTION_RECORD_INVALID`.

An approved action requires a concrete matching nonterminal job and null
execution/lifecycle/final/error fields. It requires a canonical active-review
ID and four nonzero planned hashes. A malformed action/job preflight is
`ACTION_REVIEW_INVALID` or `ACTION_RECORD_INVALID` and does not generate a
token. Complete review validation happens only after the transaction reload, so
a malformed/missing review may consume one generated token; that token is never
persisted, returned, selected back from storage, logged, or retried.

After authority materialization, capture one clock instant, generate one token,
then enter the transaction. Reload action/job/review and re-evaluate in this
exact order:

1. current definitive scope/policy block;
2. complete active review structure/linkage;
3. review expiry (`expiresAt <= now` is expired);
4. current scope hash;
5. current policy hash;
6. current action hash;
7. current context hash;
8. approved-to-running CAS.

Current block is a repair path even if the linked review is otherwise
malformed: invalidate the matching review when present, move the action to
`blocked`, clear `active_review_id` and all four planned hashes, append a
blocked event, finalize the job, commit, then throw `ACTION_SCOPE_DRIFT` or
`ACTION_POLICY_DRIFT` outside the transaction.

Expiry or hash drift invalidates the review once, moves the action to `pending`,
clears `active_review_id` and all four planned hashes, appends a paused event,
finalizes the job, commits, then throws outside the transaction:

- expiry -> `ACTION_REVIEW_EXPIRED`;
- scope -> `ACTION_SCOPE_DRIFT`;
- policy -> `ACTION_POLICY_DRIFT`;
- action or context -> `ACTION_HASH_MISMATCH`.

The persisted invalidation reason and matching rejection-event `reasonCode`
come from this fixed allowlist only:

- expiry -> `review_expired`;
- scope drift -> `scope_drift`;
- policy drift -> `policy_drift`;
- action hash drift -> `action_hash_mismatch`;
- context hash drift -> `context_hash_mismatch`;
- current scope block -> `scope_blocked`;
- current policy block -> `policy_blocked`.

Expiry/hash-drift invalidation must update exactly one complete linked active
review. Current-block repair may update zero reviews when the review is absent,
mismatched, or already invalidated, but the action CAS must still update exactly
one row. Any action-CAS loss rolls back a preceding review invalidation.

The callback returns a tagged success/rejection value. It never throws a domain
rejection after making the invalidation writes. Therefore COMMIT failure wins
over the domain error and leaves no partial result.

The successful CAS repeats action ID, job ID, status `approved`, active review,
all four planned hashes, and null token/owner/lease/start/dispatch/finish/
certainty/error predicates. It writes `running`, the generated token, canonical
owner, lease expiry, `started_at`, and `updated_at`; dispatch/final/error fields
remain null. Lease expiry is `min(now + leaseMs, review.expiresAt)` and must be
strictly later than now. Compute `now + leaseMs` with checked arithmetic: the
millisecond value must remain finite, a safe integer, within the JavaScript
`Date` range, survive an ISO round trip, and produce a lease strictly later than
`now`. Any failure is fixed, cause-free `ACTION_LEASE_INVALID` before the claim
CAS.

Unique-token constraint failure is fixed `ACTION_TOKEN_COLLISION`, is not
retried, and rolls back. CAS `changes !== 1` is `ACTION_CLAIM_RACE`.

Successful event:

```text
phase:   action-execution
status:  running
message: Action execution claimed
metadata keys exactly:
  actionId, reviewId, reasonCode, contextHash

reasonCode: execution_claimed
```

Rejection event:

```text
phase:   action-review
status:  paused (expiry/drift) | blocked (current block)
message: Action approval invalidated (expiry/drift)
         Action approval blocked by current authority (current block)
metadata keys exactly:
  actionId, reviewId, reasonCode
```

Job state is always owned by `JobManager.finalizeInTransaction`. Do not write a
special `policy_drift` pause reason outside the finalizer; a demoted pending
required action derives `approval_required`.

## 5. Dispatch mark

After pure validation and one clock capture, the transaction requires:

1. action exists and status is `running`;
2. bound token equality;
3. exact canonical owner equality;
4. canonical active-review/lifecycle fields;
5. a null dispatch marker;
6. a well-formed active lease and still-valid review;
7. CAS with the exact stored lease value.

Status mapping follows claim. A second mark has precedence over later lease
checks and always throws `ACTION_DISPATCH_ALREADY_MARKED`. Token mismatch is
`ACTION_DISPATCH_TOKEN_MISMATCH`; owner mismatch is `ACTION_OWNER_INVALID`;
malformed lease is `ACTION_LEASE_INVALID`; expired lease/review is
`ACTION_LEASE_EXPIRED`; CAS loss is `ACTION_DISPATCH_RACE`.

For `markDispatch` and `finalize`, every recognized pre-running status
(`pending`, legacy `planned`, or `approved`) maps to `ACTION_NOT_APPROVED`
before token or owner comparison. Only a `running` row reaches the bound-token
checks; `outcome_unknown` maps to `ACTION_RECONCILIATION_REQUIRED`, and
`executed | blocked | failed` maps to `ACTION_TERMINAL`.

The CAS writes only `dispatch_started_at` and `updated_at`. No workflow event or
job update is added for the marker. The method returns only the canonical
dispatch timestamp.

## 6. Finalize

All finalize outcomes require a running row, exact bound token, exact owner,
well-formed active lease, valid active review, and CAS against the exact stored
lease. An expired lease rejects with `ACTION_LEASE_EXPIRED`; recovery owns the
terminal transition. Status/token/owner errors use the mapping above.

Outcome mapping is exact:

- `success` without marker -> `ACTION_DISPATCH_NOT_MARKED`, zero writes;
- `success` with marker -> `executed/success`;
- `not_dispatched` without marker -> `failed/not_dispatched`;
- `not_dispatched` with marker -> `outcome_unknown/possibly_dispatched`;
- `possibly_dispatched` with or without marker ->
  `outcome_unknown/possibly_dispatched`.

The terminal CAS clears token, owner, and lease; retains the original active
review and four planned hashes; preserves `started_at` and marker; writes
`finished_at` and `updated_at`; and writes `executed_at` only for success.
Failure code/message are sanitized and bounded before persistence. Success
clears last error fields. CAS loss is `ACTION_FINALIZE_RACE`.

Event mapping:

- executed: `action-execution/completed`, `Action execution completed`;
- failed: `action-execution/failed`, `Action execution failed`;
- outcome unknown: `action-execution/paused`,
  `Action execution outcome is unknown`.

Metadata keys are exactly `actionId`, `status`, `outcomeCertainty`, and
`reasonCode`. The reason is respectively `effect_succeeded`, `not_dispatched`,
or `reconciliation_required`. Error
text, token, owner, target, metadata, and raw authority are forbidden. Event and
job finalization are in the same transaction as the action CAS.

## 7. Expired-lease recovery

Recovery validates only the canonical action ID. A token-free preflight may
return `not_running` without calling the clock. A path that observes `running`
captures one clock and owns one short transaction, where status is rechecked.

- Missing action -> `ACTION_NOT_FOUND`.
- Non-running -> `{ kind: "not_running" }`, no writes.
- A well-formed lease strictly later than now -> `{ kind: "active" }`, no
  writes, even if a dispatch marker exists.
- Once expired, a fully well-formed running row with no dispatch marker becomes
  `failed/not_dispatched`.
- An expired row with a marker, or any row whose token/owner/lease/start state
  is missing or malformed so pre-dispatch cannot be proven, becomes
  `outcome_unknown/possibly_dispatched`.

Recovery never reads the token value, retries, reclaims, returns to pending or
approved, or invokes an effect. It clears token/owner/lease, sets finish/update
timestamps and fixed token-free error text, appends one event, finalizes the
job, and returns the recovered result after commit. Pre-dispatch recovery uses
`action-recovery/failed`, message `Expired action lease recovered before
dispatch`, and reason `lease_expired_not_dispatched`. Ambiguous recovery uses
`action-recovery/paused`, message `Expired action lease requires
reconciliation`, and reason `reconciliation_required`. Metadata keys are
exactly `actionId`, `status`, `outcomeCertainty`, and `reasonCode`. Repeated
recovery returns `not_running`.

If the recovery terminal CAS changes zero rows after the locked running-row
classification, it throws `ACTION_FINALIZE_RACE` inside the transaction and
rolls back all preceding state. Recovery persists the fixed code
`ACTION_LEASE_EXPIRED` with the corresponding fixed recovery message; it never
persists the caller's token or owner.

## 8. Human reconciliation

Attestation is sanitized as specified in section 2, trimmed, and must contain
1..2,000 Unicode code points after sanitization. After removing every literal
`[REDACTED]` marker, at least one non-whitespace code point must remain; a
token-only attestation is not evidence. Failure is
`ACTION_RECONCILIATION_ATTESTATION_REQUIRED`. The sanitized value is the only
attestation persisted.

The transaction requires `outcome_unknown`, the original active approval
review, and exact four-hash linkage. Review expiry or a historical invalidation
does not block reconciliation because this record proves provenance and does
not authorize an effect; malformed linkage, provenance, or hashes still fails
closed. It inserts one non-active reconciliation
review through a dedicated `ActionReviewStore` in-transaction primitive:

- decision `executed | failed`;
- source `human`;
- validated reviewer;
- sanitized attestation as note;
- `created_at === reviewed_at`;
- expiry and invalidation fields null;
- four hashes copied from the original approval.

`actions.active_review_id` remains the original approval review. The CAS maps:

- `effect_confirmed` -> `executed/success`, set `executed_at`;
- `no_successful_effect_confirmed` -> `failed/possibly_dispatched`.

Both retain the original start/dispatch/finish audit timestamps and planned
hashes. Effect-confirmed clears error fields; no-success preserves sanitized
ambiguity diagnostics. Event metadata keys are exactly `actionId`,
`reconciliationReviewId`, `resolution`, and `status`; no attestation is emitted.
The event is `action-reconciliation/completed` for effect-confirmed or
`action-reconciliation/failed` for no-success, with fixed message `Action
outcome reconciled`. The event and job finalizer share the transaction. CAS loss is
`ACTION_RECONCILIATION_RACE`. The method performs zero effects.

An action that was not observed as `outcome_unknown` maps by current status:
running -> `ACTION_LEASE_HELD`; pending/approved -> `ACTION_NOT_APPROVED`;
executed/blocked/failed -> `ACTION_TERMINAL`. If an outside preflight observed
`outcome_unknown` but the transaction loses the exact CAS, the error is
`ACTION_RECONCILIATION_RACE`.

## 9. Runtime lock

`Runtime` exposes `actionApproval: ActionApprovalService` and
`actionLifecycle: ActionLifecycle`, both on the existing database handle. It
captures the initially selected `LoadedProgram.programFile` and calls
`loadProgramFile(capturedProgramFile)` afresh for every materialization. It does
not rediscover a program through mutable `process.cwd()` and never uses cached
`runtime.config` as authority.

The runtime resolver may classify only an explicitly allowlisted trusted,
built-in registered adapter with an exact registered capability as
`{ kind: "builtin" }`; it must not infer builtin status from `defaultEnabled`,
adapter type, or missing execution configuration. Packet 2 initially allows
only the bundled `playwright` registration. Unknown adapters,
unknown/mismatched capabilities, and optional external/MCP/process-backed
integrations fail closed until a complete safe execution-policy projection is
available. Runtime wiring imports no executor/effect module and performs no
effect.

## 10. Added fixed error codes

The following close caller-shape/race gaps left unnamed by the parent contract:

- `ACTION_LIFECYCLE_INVALID`
- `ACTION_OUTCOME_INVALID`
- `ACTION_RECONCILIATION_RESOLUTION_INVALID`
- `ACTION_DISPATCH_RACE`

Messages are fixed and cause-free. Existing parent-contract codes retain their
meaning and precedence.

## 11. RED writer slices

### C1 - claim and committed rejection

1. hostile DTO/owner/lease/token-generator/clock tables produce zero SQL or
   zero writes at their pinned boundaries;
2. happy claim verifies token shape, lease cap, raw row, token-free public
   action/event, and job derivation;
3. two real database handles yield one claim winner;
4. token collision is not retried and rolls back;
5. running active/expired rows are never reclaimed;
6. expiry and scope/policy/action/context drift commit review invalidation plus
   pending/blocked transition before throwing;
7. COMMIT failure suppresses the domain rejection and rolls back;
8. loader/resolver/materialization faults produce zero writes/effects;
9. no effect/network/browser/MCP/process imports.

### C2 - dispatch, finalize, recovery

1. wrong token/owner/status/lease and missing marker produce zero writes;
2. two dispatch marks produce one permit;
3. full finalize mapping table, timestamps, certainty, error sanitizer, and
   event/job atomicity;
4. fault injection at event/job/COMMIT rolls everything back;
5. recovery active/no-op, expired pre-dispatch, expired post-dispatch,
   malformed running, and repeated no-op;
6. token is absent from raw-selected JavaScript rows, returned generic records,
   events, errors, and serialized outputs.

### C3 - reconciliation and runtime

1. policy reviewer, hostile reviewer, empty/token-only or unbounded attestation,
   and invalid resolution fail with zero writes;
2. both reconciliation mappings persist a non-active human record and no
   effect;
3. two real handles produce one reconciliation result;
4. event/job/COMMIT faults roll back review/action/event/job;
5. runtime uses fresh program loading, exact database handle, conservative
   resolver behavior, secure token generator, and imports no effect module.
